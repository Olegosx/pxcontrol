# Модуль «Шлюз Telegram»

> Статус: 🟢 реализовано (в объёме текущих функций) · Обновлено: 2026-07-17 · Владелец: olegosx

## Назначение

Единая точка взаимодействия с Telegram **поверх двух транспортов** — Bot API
и MTProto ([ADR-0007](../02-architecture/decisions/0007-hybrid-botapi-mtproto.md)).
Весь остальной код обращается к Телеграму только через этот модуль
(`TelegramGateway`) и не знает, какой транспорт использован.

## Роли транспортов ([ADR-0011](../02-architecture/decisions/0011-publish-via-userbot.md))

- **MTProto** (`Telethon`, userbot на отдельном аккаунте) — **публикация
  всего контента** (`publish`: текст/фото/видео/аудио/документ, сразу или
  отложенно через `schedule_date`), чтение отложенных (`get_scheduled`);
  в будущем — чтение каналов-источников. Подключается при старте движка
  и сразу после входа; повторная активация (`activate_userbot`) закрывает
  прежнего клиента и применяет новые реквизиты — вход во второй аккаунт
  и повторный вход после отзыва сессии работают без перезапуска.
  Неудача подключения (нет сети, сессия отозвана) не мешает запуску
  приложения.
- **Bot API** (`aiogram`, по токену на операцию, без постоянного
  соединения) — проверка токена (`getMe`), проверка канала и прав бота
  (`getChat`/`getChatMember`), диагностика «где состоит бот»
  (`getUpdates`) и **запасной транспорт публикации** для каналов без
  userbot-админа: `send_text` и `send_media` (текст и медиа до 50 МБ,
  только «сейчас»). Маршрутизация — в `PostsService`
  по `publish_capabilities`.

## Устройство

- `gateway.py` — фасад `TelegramGateway`; сервисы объявляют узкие
  протоколы нужной им части (разделение интерфейсов).
- `types.py` — `OutgoingPost`: единая сущность исходящего поста на
  границе «сервисы → транспорты».
- `refs.py` — разбор пользовательского ввода «какой канал»
  (`normalize_chat_ref`: @имя, ссылки t.me/…, числовой ID) — общий
  модуль обоих транспортов; ошибка разбора — `ChatRefError`.
- `mtproto.py` — `MtprotoTransport` (публикация, чтение) и
  `MtprotoLoginManager` (пошаговый вход: код → 2FA → строка сессии).
- `bot_api.py` — функции Bot API (aiogram).

## Ошибки userbot (таксономия)

Ошибки Telethon переводятся в доменные исключения с понятным текстом;
подклассы `UserbotUnavailableError` разводят причины, потому что сервисы
принимают по ним разные решения:

- `UserbotNotConnectedError` — не подключён / нет связи (временное);
- `UserbotSessionExpiredError` — сессия отозвана, нужен повторный вход;
- `UserbotAccessError` — подтверждённый отказ Telegram: нет прав
  или канал не виден. **Только** этот класс — основание менять хранимый
  флаг `userbot_admin` при перепроверке доступов (сбой сети — нет).

## Отложено

- Медиагруппы (альбомы), кнопки.
- Повторные попытки и учёт лимитов (`FloodWait` показывается человеку).
