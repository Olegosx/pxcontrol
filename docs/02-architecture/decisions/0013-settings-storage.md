# ADR-0013. Хранение настроек: строка «владелец → имя → значение» + реестр ключей

> Статус: accepted
> Дата: 2026-07-16 · Автор: olegosx
>
> Реализует пункт «конфигурация в БД» [ADR-0009](0009-config-secrets-in-db.md).

## Контекст

Настройки приложения (тема, состояние окна, путь к ffmpeg, последний
канал публикации…) не хранились вообще: зарезервированная таблица
`settings` пустовала, тема сбрасывалась при перезапуске. Настроек будет
прибавляться, поэтому нужна расширяемая схема: новая настройка не должна
требовать миграции. При этом «ключ-значение» без дисциплины вырождается
в свалку: мусорные имена, непроверяемые типы, осиротевшие строки.

## Решение

1. **Хранение — «строка = имя → значение», таблица на вид владельца**:
   `app_settings (name PK, value JSON)` — настройки приложения;
   `channel_settings (channel_id FK ON DELETE CASCADE, name, value,
   PK(channel_id, name))` — настройки канала. Новая сущность-владелец —
   таблица-близнец по тому же шаблону. Отдельная таблица на владельца
   (вместо одной полиморфной «scope + scope_id») сохраняет настоящий
   внешний ключ и каскад: настройки живут и умирают вместе с сущностью.
2. **Состав, типы и умолчания задаёт реестр ключей в коде**
   (`engine/services/settings.py`): каждая настройка объявляется
   объектом `SettingKey` (имя, владелец, умолчание, тип значения).
   Сервис принимает **только объекты ключей** — мусорные имена
   не заводятся в принципе. Значение из БД валидируется по ключу;
   битое — откат к умолчанию с предупреждением в логе. Ключи
   типизированы (generic) — доступ проверяет mypy.
3. **`SettingsService`**: `get`/`set` (приложение),
   `get_for`/`set_for` (канал); настройки приложения кэшируются
   при старте (`prime`) — `cached()` даёт синхронный доступ
   для провайдеров (например, путь к ffmpeg в блокирующих участках).
4. **Разграничение хранилищ**: идентичность и связи сущностей —
   колонки их таблиц; секреты — только шифрованные колонки (ADR-0009);
   в `.env` — только бутстрап до открытия БД (`DATABASE_URL`,
   `LOG_LEVEL`). `FFMPEG_PATH` переезжает в настройки (правится
   из интерфейса, отдаётся сервисам **провайдером** — смена без
   перезапуска; пусто — прежний бутстрап из `.env`/PATH).
5. Колонка-заделка `channels.video_preset_id` (потребителя не было)
   удалена — её роль займёт настройка канала `default_video_preset`.

## Альтернативы

- **Колонки в таблице-одиночке** (`app_settings` с колонкой
  на настройку). Типы держит СУБД, но каждая настройка — миграция,
  а таблица из одной строки — особый случай, выбивающийся из схемы.
  Отклонено: настроек будет много и часто.
- **Одна полиморфная таблица** (`scope`, `scope_id`, `name`, `value`).
  Меньше таблиц, но внешний ключ невозможен — осиротевшие строки
  и ручная чистка. Отклонено.
- **QSettings/INI для интерфейса**. Родной механизм Qt, но появляется
  третье хранилище и нарушается ADR-0009 («конфигурация — в БД»,
  портативность одним файлом). Отклонено.

## Последствия

- **Плюсы:** новая настройка = ключ в реестре (без миграций); типы
  и умолчания в одном месте; целостность и каскад — на СУБД; один
  механизм для приложения и сущностей.
- **Минусы/риски:** тип значения проверяет код, а не СУБД (размен
  на расширяемость); JSON-значения не для сложных структур — если
  «настройка» обрастает связями, ей место в нормальной таблице.

## Дополнение (2026-07-16, завершение)

- Колонка-заделка `channels.enabled` также переехала настройкой канала
  `enabled` (миграция `c8f1d29e4a35`; переносятся только выключенные —
  в БД хранятся отличия от умолчаний). Потребители: переключатель
  в карточке канала; выключенный канал не показывается на «Публикации»
  и не опрашивается «Расписанием».
- `default_video_preset` получил потребителей: кнопка «Пресет…»
  в карточке канала и выбор канала на странице «Видео» (подставляет
  пресет в панель параметров).
- **Настройки-ссылки** (id пресета): внешнего ключа у значения нет,
  целостность держит сервис («вариант а») — `VideoService.delete_preset`
  снимает удалённый пресет у каналов (`drop_channel_value`), битая
  ссылка читается как «не задано».
- Тип значения проверяется и при **записи** (`SettingsError`),
  а не только при чтении; чтение по-прежнему мягкое — откат к умолчанию.
