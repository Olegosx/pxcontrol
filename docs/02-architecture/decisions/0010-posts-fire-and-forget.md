# ADR-0010. Посты: источник истины — канал, доставка — сервер Telegram

> Статус: accepted
> Дата: 2026-07-12 · Автор: olegosx
>
> Уточняет [ADR-0004](0004-apscheduler-tasks.md) и [ADR-0007](0007-hybrid-botapi-mtproto.md).

## Контекст

Первоначальный план хранил посты в локальной таблице и публиковал их
собственным планировщиком (APScheduler) с «догоном» пропущенного. Но:

1. Содержимое канала меняется независимо от приложения (посты правят
   и удаляют из любого клиента Telegram) — локальная таблица постов
   неизбежно расходится с реальностью и требует синхронизации.
2. MTProto поддерживает **серверное планирование** (`schedule_date`):
   отложенная запись создаётся прямо в канале, хранит и публикует её
   сервер Telegram — приложение и компьютер могут быть выключены.

## Решение

1. **Источник истины по постам — сам канал.** Локальной таблицы постов
   нет. Страница «Расписание» читает отложенные записи из Telegram
   (`getScheduledHistory`). Приложение работает по принципу
   fire-and-forget: создал → отправил → забыл.
2. **«Сейчас»** публикует **бот** (Bot API) — безопаснее для аккаунта.
3. **Отложенные** создаёт **userbot** (`schedule_date`) — публикацию
   выполняет сервер Telegram; править/удалять отложенные можно из
   любого клиента Telegram.
4. Наш планировщик и «догон» постам больше не нужны; APScheduler
   остаётся в движке для будущего парсинга источников (ADR-0004).
5. Userbot подключается при старте движка (если в БД есть сессия)
   и сразу после успешного входа.

## Альтернативы

- **Локальная таблица + свой планировщик.** Полный контроль и история,
  но второй источник истины, синхронизация и «догон». Отклонено на этом
  этапе; таблица постов вернётся, когда появится реальная нужда
  (очередь модерации, сборка видео-поста).

## Последствия

- **Плюсы:** нет рассинхронизации; отложки переживают выключенный ПК;
  видны и редактируются из любого клиента Telegram; меньше механики.
- **Минусы / риски:** создание отложек — действие от имени аккаунта
  userbot (риск умеренный на наших объёмах; отдельный аккаунт);
  **userbot должен быть администратором каждого канала** с правом
  публиковать; локальной истории публикаций нет (ошибки — в логе).
