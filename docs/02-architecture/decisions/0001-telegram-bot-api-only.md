# ADR-0001. Telegram только через Bot API

> Статус: superseded — заменено [ADR-0007](0007-hybrid-botapi-mtproto.md)
> Дата: 2026-06-30 · Автор: olegosx
>
> ⚠️ Решение пересмотрено: подключаем гибрид Bot API + MTProto (нужен парсинг
> чужих каналов, которого Bot API не даёт). Запись сохранена как история
> и как перечень рисков MTProto. См. [ADR-0007](0007-hybrid-botapi-mtproto.md).

## Контекст

Приложению нужно публиковать контент в Telegram-каналы и управлять ими.
У Telegram есть два пути взаимодействия:

- **Bot API** — официальный HTTP-интерфейс для ботов. Бот становится
  администратором канала и публикует посты.
- **MTProto (userbot)** — работа от имени обычного пользователя через
  библиотеки Telethon/Pyrogram. Даёт полный доступ, включая чтение истории
  любых каналов.

## Решение

Используем **только Bot API**. От MTProto/userbot отказываемся.

## Альтернативы

- **MTProto / userbot.**
  - Плюсы: полный доступ, возможность читать историю чужих каналов,
    действия от имени пользователя.
  - Минусы: высокий риск блокировки аккаунта Telegram, сложнее в поддержке,
    серая зона по правилам платформы.
  - Отклонено: риск блокировки и юридическая серость перевешивают.
- **Гибрид Bot API + MTProto.**
  - Плюсы: максимум возможностей.
  - Минусы: те же риски userbot плюс удвоенная сложность поддержки.
  - Отклонено на старте; при необходимости парсинга закрытых источников
    вопрос можно пересмотреть отдельной записью ADR.

## Последствия

- **Плюсы:** стабильность, безопасность аккаунта, простота, соответствие
  правилам Telegram.
- **Минусы / риски:** нельзя читать историю произвольных каналов через
  бота; парсинг внешних каналов как источников ограничен возможностями
  Bot API. Источники-каналы, возможно, придётся подключать иначе
  (например, через публичные веб-версии или RSS-мосты) — уточнить в
  [content-sources.md](../../03-modules/content-sources.md).
- **Что дальше:** проверить ограничения Bot API на действия, нужные для
  сценариев из [use-cases.md](../../01-overview/use-cases.md).
