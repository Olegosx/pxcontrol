# ADR-0002. Веб-интерфейс с серверным бэкендом

> Дата: 2026-06-30 · Автор: olegosx
>

## Контекст

Нужно выбрать, через что пользователь управляет приложением. Рассматривались
веб-приложение (браузер) и десктопное приложение. Ключевое ограничение:
часть функций работает **без участия человека и круглосуточно** —
отложенный постинг по расписанию и регулярный парсинг источников. Значит,
нужен процесс, который крутится постоянно, независимо от того, открыт ли
интерфейс.

## Решение

Делаем **веб-приложение**: серверный бэкенд на Python с круглосуточным
фоновым обработчиком плюс веб-панель в браузере.

## Альтернативы

- **Десктопное приложение.**
  - Плюсы: работа без публичного сервера, локальное хранение данных.
  - Минусы: запущено только пока включён компьютер пользователя —
    ночной отложенный пост не выйдет, парсер не отработает. Чтобы это
    починить, всё равно нужен серверный компонент, и десктоп вырождается
    в ещё один экран поверх сервера.
  - Отклонено: не закрывает требование круглосуточной работы.
- **CLI / скрипты.**
  - Плюсы: быстро для технического пользователя.
  - Минусы: неудобно для модерации очереди и визуального планирования.
  - Отклонено как основной интерфейс (может остаться вспомогательным).

## Последствия

- **Плюсы:** серверная часть нужна всё равно — веб-панель поверх неё даёт
  наименьшие дополнительные трудозатраты; доступ из любого браузера.
- **Минусы / риски:** нужен хостинг и забота о безопасности веб-доступа
  (аутентификация, HTTPS). Уточнить в
  [deployment.md](../../07-operations/deployment.md).
- **Что дальше:** выбрать веб-фреймворк и подход к фронтенду в
  [tech-stack.md](../tech-stack.md); утвердить эту запись (перевести из
  proposed в accepted).
