## Контекст

Нужен механизм для трёх видов фоновой работы: публикация постов в заданное
время (отложенный постинг), регулярный опрос источников (парсинг) и разовые
задания генерации ИИ с повтором при сбое. Всё это должно работать
круглосуточно, независимо от интерфейса.

## Решение

Используем **APScheduler 3.x** с хранением задач в базе данных (jobstore
на SQLAlchemy), чтобы задачи переживали перезапуск процесса.

## Альтернативы

- **Celery + Redis (+ beat).**
  - Плюсы: зрелый распределённый брокер, повторы, масштабирование на
    несколько воркеров.
  - Минусы: требует брокер (Redis) и дополнительную инфраструктуру;
    синхронный по природе; избыточен для одного экземпляра.
  - Отклонено на старте (YAGNI).
- **arq (async + Redis).**
  - Плюсы: лёгкий, асинхронный, есть cron-задачи.
  - Минусы: всё равно требует Redis.
  - Отклонено на старте; основной кандидат при миграции (см. ниже).

## Последствия

- **Плюсы:** ноль дополнительной инфраструктуры (брокер не нужен) —
  меньше движущихся частей для MVP; работает в процессе фонового
  обработчика.
- **Минусы / риски:** APScheduler в одном процессе не масштабируется
  горизонтально на несколько воркеров. Для одного экземпляра это
  приемлемо.
- **Путь миграции:** если понадобится несколько воркеров или высокий поток
  задач — переходим на **arq + Redis** (ближе по духу к async-стеку, чем
  Celery), оформив новой записью ADR.
