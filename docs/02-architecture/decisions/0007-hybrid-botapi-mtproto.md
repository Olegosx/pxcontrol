# ADR-0007. Гибрид Telegram: Bot API + MTProto

> Статус: accepted
> Дата: 2026-06-30 · Автор: olegosx
>
> Заменяет: [ADR-0001](0001-telegram-bot-api-only.md).
>
> ⚠️ Ориентир по постингу уточнён ADR-0011: публикация любого контента —
> через userbot (MTProto), Bot API — запасной путь для каналов без
> userbot-админа (см. [ADR-0011](0011-publish-via-userbot.md)).

## Контекст

В [ADR-0001](0001-telegram-bot-api-only.md) мы сознательно ограничились
Bot API, чтобы не рисковать блокировкой аккаунта. Но это закрывает важную
функцию из видения — **парсинг чужих Telegram-каналов как источников
контента**: Bot API не даёт читать историю произвольных каналов. Дополнительно
MTProto умеет **серверное планирование** (`schedule_date`) — пост уходит,
даже если приложение выключено.

Решено пересмотреть запрет MTProto, но осознанно, с учётом рисков.

## Решение

Подключаем **оба транспорта** к Telegram:

- **Bot API** через `aiogram` — публикация и управление каналами.
- **MTProto (userbot)** через `Telethon` — чтение/парсинг каналов, при
  необходимости серверное планирование.

**Жёсткой границы между ними не вводим** — инструмент выбирается по
целесообразности в конкретной операции. Ориентир (не правило): постинг
тяготеет к Bot API (безопаснее для аккаунта), чтение источников — к MTProto.

Userbot работает на **отдельном аккаунте** (не на личном основном).

## Альтернативы

- **Только Bot API (ADR-0001).** Безопасно, но не даёт парсить каналы.
  Отклонено — функция нужна.
- **Полностью на MTProto.** Максимум возможностей, но максимум риска бана
  на самой частой операции (постинге). Отклонено — постинг безопаснее через
  бота.

## Последствия

- **Плюсы:** открывается парсинг Telegram-каналов; доступно серверное
  планирование как опция; гибкость выбора транспорта.
- **Минусы / риски:**
  - **Риск блокировки аккаунта** за автоматизацию. Митигаторы: отдельный
    аккаунт; постинг по возможности через Bot API; соблюдать лимиты частоты;
    избегать «спамных» действий.
  - **Безопасность сессии.** MTProto хранит строку сессии (session string),
    равную по силе паролю аккаунта — шифруем локально, не кладём в код и не
    логируем (см. [configuration.md](../../06-config/configuration.md)).
  - Сложнее: логин по номеру телефона, 2FA, второй клиент в коде.
- **Что дальше:** спроектировать шлюз Телеграма как единую точку над двумя
  транспортами ([telegram-gateway.md](../../03-modules/telegram-gateway.md)).
