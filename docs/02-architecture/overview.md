# Обзор архитектуры

> Статус: 🟢 актуально · Обновлено: 2026-07-17 · Владелец: olegosx

## Назначение документа

Дать общую картину системы «с высоты птичьего полёта»: какие крупные
части есть, как они связаны, где проходят границы. Детали — в отдельных
документах по компонентам и модулям.

## Контекст

Кто и что взаимодействует с системой снаружи:

- **Пользователь** → десктопное приложение (нативное окно).
- **Приложение** → Telegram: гибрид MTProto (userbot — публикация
  и чтение, приоритетный транспорт, [ADR-0011](decisions/0011-publish-via-userbot.md))
  + Bot API (проверки, диагностика и запасная публикация,
  [ADR-0007](decisions/0007-hybrid-botapi-mtproto.md)).
- **Приложение** → внешние источники (сайты, RSS, каналы) для парсинга (план).
- **Приложение** → провайдер ИИ (например, Claude API) для генерации (план).

## Крупные части системы

1. **Интерфейс (PySide6 + QFluentWidgets)** — нативное окно приложения;
   к движку обращается только через мост (корутина в цикле движка,
   результат — сигналом Qt).
2. **Движок (engine)** — UI-независимый Python-пакет с бизнес-логикой;
   интерфейс вызывает его напрямую в процессе.
3. **Фоновый поток движка** — свой цикл событий asyncio: сеть, БД,
   очередь отправки; кодирование видео и другие блокирующие операции
   уходят из цикла в отдельные потоки (`asyncio.to_thread`). Вынесен
   в поток, чтобы окно не зависало.
4. **Хранилище** — SQLite: каналы, аккаунты, пресеты видео, подписи,
   настройки. Постов и очереди в БД нет: отложенные хранит и публикует
   сервер Telegram ([ADR-0010](decisions/0010-posts-fire-and-forget.md)),
   очередь отправки живёт в памяти ([ADR-0012](decisions/0012-publish-queue-in-memory.md)).

> ⚠️ Своего планировщика в приложении нет: отложенные посты создаются
> прямо в канале (`schedule_date`), их публикует сервер Telegram — даже
> когда приложение закрыто. Страница «Расписание» читает отложенные
> из Telegram (источник истины — сам канал, ADR-0010).
